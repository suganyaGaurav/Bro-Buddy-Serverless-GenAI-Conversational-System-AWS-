# Backend — Bro-Buddy AI Companion System

## Overview

This folder contains the **backend architecture for Bro-Buddy**, a governance-first AI companion system designed with safety, routing control, and observability.

The backend is implemented as a **serverless pipeline running on AWS Lambda**, where each module performs a specific responsibility such as safety filtering, routing, orchestration, and response validation.

The system interacts with an LLM hosted on **AWS Bedrock (Claude)** to generate responses while enforcing strict safety and governance layers.

---

# Backend Architecture

The system follows a **layered processing pipeline** to ensure that user input is validated, secured, routed correctly, and safely returned.

Pipeline Flow:

```
config → handler → capacity_guard → privacy_guard → firewall → routing → memory → orchestrator → llm_client → prompt → validator → logging_utils
```

Each component plays a role in maintaining **safety, cost control, reliability, and observability**.

---

# Core Modules

## handler.py

Entry point for the AWS Lambda function.

Responsibilities:

* Receives incoming requests from API Gateway
* Validates request structure
* Initializes the pipeline
* Coordinates request processing
* Returns the final response

This file acts as the **main orchestrator trigger for the backend pipeline**.

---

## config.py

Central configuration module for system settings.

Responsibilities:

* Stores environment configuration
* Defines model parameters
* Maintains system constants
* Provides reusable configuration values across modules

---

# Guardrail Layers

The system uses multiple guardrails before interacting with the AI model.

These layers enforce **AI governance and safety policies**.

---

## capacity_guard.py

Prevents excessive traffic and protects LLM usage costs.

Responsibilities:

* Tracks concurrent requests
* Prevents request flooding
* Protects Bedrock usage limits
* Returns graceful responses when limits are exceeded

---

## privacy_guard.py

Protects user privacy by detecting and masking personally identifiable information (PII).

Responsibilities:

* Detect email addresses
* Mask sensitive user information
* Prevent PII from entering prompts

Example PII types handled:

* email
* phone numbers
* identifiers

---

## firewall.py

Security layer designed to detect **prompt injection and system probing attempts**.

Responsibilities:

* Detect architecture probing
* Detect identity probing
* Detect attempts to extract system prompts
* Block malicious requests

Example blocked behaviors:

* system prompt extraction
* jailbreak attempts
* architecture probing

---

# Routing Layer

## routing.py

Determines how the system processes the user query.

Responsibilities:

* Intent classification
* Deterministic routing
* Greeting shortcuts
* Mode-based routing

Supported modes:

* Professional Mode
* AI Mode

This module decides whether the request should:

* return a deterministic response
* call the LLM
* trigger safety responses

---

# Memory Layer

## memory_store.py

Handles conversational memory for maintaining session context.

Responsibilities:

* Store conversation history
* Retrieve session context
* Maintain continuity between messages

Memory is session-based and scoped per user interaction.

---

# Orchestration Layer

## orchestrator.py

Coordinates the full AI response pipeline.

Responsibilities:

* Combine routing results
* Prepare prompts for the LLM
* Manage execution flow
* Handle interaction between system components

This module ensures the system behaves as a **cohesive AI workflow** rather than isolated modules.

---

# AI Interaction Layer

## llm_client.py

Handles communication with the LLM hosted on AWS Bedrock.

Responsibilities:

* Send prompts to the model
* Receive generated responses
* Track response latency
* Capture output metrics

The system currently integrates with:

* Claude models via AWS Bedrock

---

## prompt.py

Defines the prompt structure used for model interaction.

Responsibilities:

* Maintain system identity
* Enforce safety policies
* Define system instructions
* Format prompts for the LLM

This ensures consistent behavior across model interactions.

---

# Response Safety Layer

## validator.py

Ensures that responses generated by the LLM comply with system safety rules.

Responsibilities:

* Prevent prompt leakage
* Remove infrastructure details
* Filter unsafe responses
* Enforce response size limits

This layer acts as a **final safeguard before returning responses to users**.

---

# Observability and Logging

## logging_utils.py

Provides structured logging across the system.

Responsibilities:

* Record pipeline events
* Track request lifecycle
* Capture latency metrics
* Support debugging and monitoring

Example logged events:

* request_received
* intent_classified
* routing_decision
* llm_request_started
* llm_request_completed
* memory_updated

These logs enable **traceability and system observability**, which are critical for production AI systems.

# Design Principles

The backend was built using the following principles:

Safety First
Multiple guardrail layers protect the AI system from misuse.

Deterministic Routing
Simple queries are handled without calling the LLM to reduce cost and latency.

Observability
Structured logging ensures full visibility into system behavior.

Modular Architecture
Each module performs a clear responsibility, improving maintainability.

Governance-Focused AI
The architecture prioritizes safety, transparency, and responsible AI behavior.

# Deployment Architecture

The backend is deployed using a **serverless architecture**:

```
Frontend UI
     ↓
API Gateway
     ↓
AWS Lambda (Bro-Buddy Backend)
     ↓
AWS Bedrock LLM
     ↓
CloudWatch Logs
```

This design ensures scalability, reliability, and minimal infrastructure management.
